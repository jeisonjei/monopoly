# Простые Python-задачи для ребёнка-новичка

В этом проекте Python-бэкенд находится в папке `server/`.
Ниже собраны небольшие и безопасные места, где ребёнок 11 лет, который только учит Python, может попробовать что-то исправить или улучшить.

## Подходящие области для начинающего

### 1. `server/game/tests.py`

Текущее состояние:
- Файл почти пустой.
- В нём только стандартная заглушка для Django-тестов.

Почему это хороший вариант:
- Писать простые тесты очень полезно для новичка.
- Это безопасно, потому что тесты не меняют игровую логику.

Возможные задачи для ребёнка:
- Добавить тест, который проверяет, что `get_tile_definition(0)` возвращает клетку `go`.
- Добавить тест, который проверяет, что `is_ownable_tile(1)` равно `True`.
- Добавить тест, который проверяет, что `is_ownable_tile(2)` равно `False`.
- Добавить тест, который проверяет, что `create_special_card_payload(7)` возвращает карту chance.

Уровень сложности:
- **Отлично подходит для новичка**

---

### 2. `server/game/admin.py`

Текущее состояние:
- Файл импортирует Django admin.
- Модели пока не зарегистрированы.

Почему это хороший вариант:
- Регистрация моделей в Django admin довольно простая.
- Ребёнок потренируется работать с импортами, классами и базовой структурой Django.

Возможная задача:
- Зарегистрировать `Game`, `PlayerState` и `PropertyState` в Django admin.

Небольшое усложнение:
- Показать несколько полезных полей в списке admin.

Уровень сложности:
- **Отлично подходит для новичка**

---

### 3. `server/game/board_tiles.py`

Текущее состояние:
- В этом файле есть данные о клетках поля и простые вспомогательные функции.
- Функции небольшие и понятные.

Почему это хороший вариант:
- Логика читается легко.
- Это хорошее место, чтобы потренировать словари, функции и `if`.

Возможные задачи для ребёнка:
- Сделать так, чтобы `get_tile_definition()` использовала именованную константу для размера поля вместо числа `40`.
- Добавить небольшую функцию, например `is_special_tile(tile_index)`.
- Улучшить запасное имя клетки так, чтобы использовался нормализованный индекс.
- Добавить ещё одну простую ветку для special card payload.

Уровень сложности:
- **Хорошо подходит для новичка**

## С чего лучше начать

Если нужны самые простые первые задачи, лучше идти в таком порядке:

1. `server/game/tests.py`
2. `server/game/admin.py`
3. `server/game/board_tiles.py`
4. `server/accounts/serializers.py`
5. `server/accounts/views.py`

## Что пока лучше не трогать

Эти файлы, скорее всего, пока слишком сложные для маленького новичка:

### `server/game/consumers.py`
- Файл длинный.
- В нём используется async-код и websockets.
- Там много логики ходов, обновления состояния и разных веток условий.

### `server/game/models.py`
- Там есть асинхронные помощники для базы данных.
- Используются транзакции и ограничения моделей.
- Ошибка в этом месте может сломать основную игровую логику.

## Хорошая первая мини-задача

Очень хороший первый мини-проект для ребёнка:

1. Открыть `server/game/tests.py`
2. Импортировать функции из `server/game/board_tiles.py`
3. Написать 3 или 4 маленьких теста
4. Попросить взрослого посмотреть результат

## Как начать работу с репозиторием в GitHub

Ниже простой порядок действий, если ребёнок или взрослый хочет сделать отдельную маленькую задачу правильно.

### 1. Клонировать репозиторий

Если репозиторий уже есть в GitHub, его можно скачать на компьютер командой:

```bash
git clone <URL_РЕПОЗИТОРИЯ>
```

Пример:

```bash
git clone https://github.com/username/monopoly.git
```

После этого нужно открыть папку проекта.

### 2. Создать отдельную ветку для задачи

Новая ветка нужна, чтобы не работать прямо в основной ветке.

Пример команды:

```bash
git checkout -b child/fix-board-tiles
```

Идеи для названий веток:
- `child/add-tests`
- `child/register-admin-models`
- `child/fix-board-size-constant`

### 3. Сделать изменения

После создания ветки можно:
- изменить код
- сохранить файлы
- проверить, что всё выглядит правильно

### 4. Добавить изменения в git

```bash
git add .
```

Если хочется добавить только один файл, можно так:

```bash
git add server/game/tests.py
```

### 5. Создать коммит

```bash
git commit -m "Add simple board tile tests"
```

Лучше писать короткое и понятное сообщение.

### 6. Отправить ветку в GitHub

```bash
git push -u origin child/fix-board-tiles
```

### 7. Создать merge request / pull request в GitHub

В GitHub обычно используется название **Pull Request**.
Иногда люди по привычке говорят **Merge Request**, но в интерфейсе GitHub это именно Pull Request.

Что делать:
- открыть репозиторий на GitHub
- GitHub обычно сам предложит кнопку **Compare & pull request**
- нажать эту кнопку
- написать короткий заголовок
- написать, что именно было изменено
- создать Pull Request

Пример заголовка:
- `Add simple tests for board tiles`

Пример описания:
- Added a few beginner-friendly tests for `board_tiles.py`
- Checked ownable and special tile behavior

### 8. Попросить взрослого проверить изменения

Это особенно полезно, если ребёнок только учится.
Взрослый может:
- посмотреть код
- подсказать ошибки
- помочь улучшить названия
- объяснить, что делает каждая команда git
